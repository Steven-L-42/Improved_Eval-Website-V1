Evaluator Behavior:

  • Focus solely on conducting the evaluation without the concern for deadlines.
  • Maintain professionalism during discussion and adhere to evaluation criteria.
  • After the evaluation, discussions should be concluded or if continued kept on a constructive and objective level.
  • Normalize seeking a second opinion, especially when evaluating unfamiliar projects (42 Heilbronn is a peer-learning university).
  • Aim for high-quality standard regarding code structure and readability.
  • Share code knowledge and avoid failing for minor errors like typos (think about what people can learn from you letting them fail).
  • Prioritize fair evaluations, avoiding rigged assessments to benefit close associates (your friends).
  • If using a tester which shows errors, you need to be able to explain what the tester does and why the code fails. Otherwise that is not a fail reason.

Evaluatee Behavior:

  • Own responsibility for your project, learn from submission issues, and recognize failure as a learning opportunity.
  • Avoid fixing evaluations with biased opinions to ensure fair learning opportunities (do not book evaluations with your friends).
  • Give honest feedback to evaluators on how they conducted the evaluation.
  • Only book one evaluation for a single slot and leave enough room for the discussions (half an hour can get quite short for a thorough evaluation).
